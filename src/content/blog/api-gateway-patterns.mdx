An API gateway is the single entry point for all client requests to your backend services.
It handles cross-cutting concerns so your services don't have to. This article covers the
four patterns you'll reach for most often.

## 1. Request routing

The gateway inspects each incoming request and forwards it to the correct upstream service.
Routing can be based on the URL path, HTTP method, headers, or any combination.

```yaml
# Kong declarative config (kong.yaml)
services:
  - name: orders-service
    url: http://orders:8080
    routes:
      - name: orders-route
        paths: ["/api/v1/orders"]
        methods: [GET, POST]

  - name: inventory-service
    url: http://inventory:8080
    routes:
      - name: inventory-route
        paths: ["/api/v1/inventory"]
```

## 2. Rate limiting

Protect your services from abuse and ensure fair usage across consumers by throttling
requests at the gateway layer.

```yaml
plugins:
  - name: rate-limiting
    config:
      minute: 100
      hour: 2000
      policy: local
```

Consumer-specific limits can be set by attaching the plugin to a Consumer object rather
than the service, giving you fine-grained control per API key or JWT subject.

## 3. Authentication & authorisation

The gateway validates credentials before the request ever reaches your service — keeping
authentication logic out of your business code.

```yaml
plugins:
  - name: jwt
    config:
      secret_is_base64: false
      claims_to_verify: [exp, nbf]
      key_claim_name: iss
```

For integrations with Keycloak or another OIDC provider, use the `openid-connect` plugin
to perform token introspection or local JWT verification with JWKS.

## 4. Circuit breaking

When an upstream service degrades, a circuit breaker stops sending requests to it for a
configurable period, allowing it to recover without cascading failures.

```yaml
plugins:
  - name: circuit-breaker
    config:
      window_size: 10
      threshold: 0.5    # Open when >50% of requests fail
      timeout: 30       # Keep circuit open for 30 s
```
